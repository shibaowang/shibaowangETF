# TASK-ACCOUNT-003: 账户资金、持仓、TradeLog 与本地回放

## 收口结论

模块 3 已实现本地 TradeLog 财务回放：

- `trade_log` 是账户资金、持仓、成本、已实现盈亏的事实源。
- `account_state` 和 `position_state` 只在没有 TradeLog 时作为手动录入展示参考；一旦存在 TradeLog，主界面财务字段以回放派生表为准。
- `market_quote_cache` 只用于估值，不用于生成交易、策略结论或假资产。
- 缺少真实行情时显示估值不完整，市值、浮盈、总盈亏等字段不使用成本价冒充。
- 财务异常会停止回放、显示“财务异常”、写入 `runtime_log`，并保留上一轮成功持仓派生表。

## 数据表

输入表不被清空、不被重建：

- `trade_log`
- `market_quote_cache`
- `account_state`
- `position_state`

模块 3 新增派生表：

- `account_replay_state`
- `position_replay_state`
- `otc_position_replay_state`

成功回放时只清空并重写派生持仓表；回放失败时写入异常账户状态，不覆盖上一轮成功派生持仓。

## TradeLog 预检

回放前校验：

- 必填：时间、策略代码、动作。
- 动作只允许：`CASH`、`入金`、`出金`、`买入`、`卖出`、`分红`、`送股`、`拆分`、`合并`、`除权校准`。
- 数值字段必须是有效数字：价格、数量、金额、手续费、净现金流、本金、现金余额、总资产。
- 手续费、金额、数量不能小于 0。
- `CASH`、`入金`、`出金` 的策略代码必须为 `CASH`。
- 入金/出金净现金流不一致、现金余额审计不一致、超卖、合并超过持仓都会标记为财务异常。
- 买入/卖出必须有正数量和正金额；自动推演账务字段不会掩盖超卖、合并超过持仓等真正财务错误。

## 手动录入字段职责

TradeLog 手动录入页中，用户只需要维护业务字段：

- `time`
- `strategy_code`
- `actual_code`
- `action`
- `price`
- `quantity`
- `amount`
- `tier`
- `source`
- `fee`
- `memo`

以下账务字段由系统保存前自动维护，UI 中只作为回放结果展示：

- `net_cash_impact`
- `principal`
- `cash_balance`
- `total_assets`

保存 TradeLog 时，系统会在写入 SQLite 前统一执行金额计算和账务字段推演，覆盖这些账务字段。用户不需要手动计算现金余额、本金、净现金流或总资产。

## 保存前自动推演

保存 TradeLog 的流程固定为：

1. 提交 DataGrid 当前单元格和当前行编辑。
2. 对买入/卖出按 `amount = price * quantity` 自动计算金额。
3. 按 `time asc, id asc` 推演 `net_cash_impact`、`principal`、`cash_balance`、`total_assets`。
4. 删除用户已标记删除的数据库记录。
5. 保存当前 UI 集合中剩余的 TradeLog。
6. 执行账户回放。
7. 从数据库重读 TradeLog 并刷新 UI。

现金推演规则：

- `CASH`：使用 `cash_balance` 作为现金基线；如为空或 0 且 `amount > 0`，使用 `amount` 作为现金基线。
- `入金`：`net_cash_impact = amount - fee`，现金和本金同步增加。
- `出金`：`net_cash_impact = -(amount + fee)`，现金和本金同步减少。
- `买入`：`net_cash_impact = -(amount + fee)`，现金减少，本金不因买入变化。
- `卖出`：`net_cash_impact = amount - fee`，现金增加，已实现盈亏由回放服务计算。
- `分红`：`net_cash_impact = amount`，现金增加。
- `送股` / `拆分` / `合并` / `除权校准`：`net_cash_impact = 0`，现金不变。

`total_assets` 只使用现金加真实行情缓存可计算的持仓市值。行情缺失时仍标记估值不完整，不用成本冒充市值。

## 金额自动计算

TradeLog 页在编辑动作、价格或数量后会尝试自动计算金额；保存前也会再次统一计算。

- 仅 `买入` / `卖出` 触发。
- 条件：`price > 0` 且 `quantity > 0`。
- 公式：`amount = price * quantity`。
- 金额保留 2 位小数，不包含手续费。
- 价格和数量继续支持小数；数量如 `1000.5678` 不会被截断。
- `入金`、`出金`、`分红` 的金额由用户直接填写。
- `送股`、`拆分`、`合并`、`除权校准` 不强制自动金额。

## 闪退防护和日志

TradeLog 录入链路已增加全局异常捕获和本地崩溃日志：

- WPF UI 线程异常通过 `DispatcherUnhandledException` 记录后标记为已处理，避免可恢复异常直接退出程序。
- 非 UI 线程异常通过 `AppDomain.CurrentDomain.UnhandledException` 记录。
- 未观察到的任务异常通过 `TaskScheduler.UnobservedTaskException` 记录并标记为已观察。
- 崩溃日志写入 `%LocalAppData%\CrossETF.Terminal.UiShell.Reference\logs\crash-yyyyMMdd.log`。
- 运行日志写入 `%LocalAppData%\CrossETF.Terminal.UiShell.Reference\logs\runtime-yyyyMMdd.log`，并尽量同步写入 SQLite `runtime_log`。
- 日志包含时间、异常类型、message、stack trace、inner exception 和当前操作上下文，例如 TradeLog 保存、DataGrid 编辑提交、金额自动计算、账务推演、删除同步、账户回放。

TradeLog 保存链路的可恢复错误必须显示在窗口底部状态栏或主界面运行状态中，不能静默失败，也不能吞掉错误后假装保存成功。

## 动作规则

- `CASH`：设置现金基线；如填写本金，则同步本金基线。
- `入金`：净现金流为 `amount - fee`；如填写 `net_cash_impact`，必须非负且与公式差异不超过 0.01。
- `出金`：净现金流为 `-(amount + fee)`；如填写 `net_cash_impact`，必须非正且与公式差异不超过 0.01。
- `买入`：现金减少 `amount + fee`，数量增加，成本增加 `amount`。
- `卖出`：现金增加 `amount - fee`，按平均成本扣减持仓成本，生成已实现盈亏，禁止超卖。
- `分红`：现金增加 `amount`，持仓不变。
- `送股`：数量增加，成本不变。
- `拆分`：数量增加，成本不变。
- `合并`：数量减少，成本不变，禁止合并超过当前持仓。
- `除权校准`：`quantity` 非 0 时作为折算系数，否则折算系数为 1。

## 删除同步

TradeLog 删除按钮分两类处理：

- 未保存的新行：只从当前 UI 集合移除。
- 已保存的数据库行：记录原始 ID，保存时在同一事务中删除数据库记录。

保存时还会用“窗口加载时的 TradeLog ID 快照”和“当前 UI 剩余 ID”做同步校验，确保被用户从 UI 移除的中间行真正从数据库删除。删除后会重新回放账户，现金和持仓按剩余 TradeLog 重算。

删除历史 TradeLog 后 ID 不会重新编号，`1-18、20` 这类不连续 ID 是正常审计行为。系统禁止为了 ID 连续重排历史 ID。

## 估值规则

- 场内 ETF 使用 `market_quote_cache` 中 `market_type='ETF'` 的真实行情价格。
- 场外替代使用 `market_type='OTC'` 的真实基金净值。
- 行情匹配按策略代码、实际代码和纯数字代码匹配。
- 缺少行情时：
  - `quote_status` 显示未连接。
  - 市值、浮盈、总盈亏、收益率显示 `--` 或估值不完整。
  - 不使用成本价、手工高点或任何假价格顶替。
- 历史高点和历史 K 线失败不影响账户资金/持仓回放，只影响依赖历史高点的后续策略前置状态。

## UI 接入

- 程序启动后自动回放。
- TradeLog 保存后自动回放。
- 行情缓存更新后自动重新估值。
- 主界面 2-4 秒随机刷新只读取本地派生状态，不提供手动刷新按钮。
- 账户状态面板展示回放状态、现金余额、最新 TradeLog ID、持仓成本、持仓市值、本金、总资产、已实现盈亏、浮动盈亏和回放时间。“持仓市值”表示有真实行情的持仓按行情估值；缺行情的持仓不使用假价格。
- ETF 表格中的数量、成本、当日盈亏、持仓盈亏、盈亏率、本金占比、折算系数、折算数量来自回放派生表。
- 派生表中的 `cost_amount` 表示当前剩余总成本金额；`average_cost` 表示当前剩余平均成本单价。
- ETF 表格的“综合持仓成本”显示 `average_cost`，即 `剩余总成本金额 / 剩余总持仓数量`。该列不显示总成本金额，不用市值或现价乘数量冒充成本。场内成本来自 `position_replay_state.cost_amount`，场外替代成本来自 `otc_position_replay_state.cost_amount`，卖出后按 AccountReplayService 的平均成本扣减结果计算平均单价。
- ETF 表格的“本金占比”继续使用剩余总成本金额 / 本金；“持仓盈亏”继续使用持仓市值 - 剩余总成本金额；“盈亏率”继续使用持仓盈亏 / 剩余总成本金额。上述字段不使用平均成本单价。
- ETF 表格的“溢价率”只来自真实行情缓存中的 `price` 和 `iopv`：`(price - iopv) / iopv`。缺少 IOPV 或 IOPV 小于等于 0 时显示 `--`，不伪造 0%，不使用现价冒充 IOPV。
- ETF 表格列名使用“ETF高点”，不使用旧的“ETF隐点”。该列显示 ETF 高点口径，不与实时估值 IOPV 混用。
- ETF 表格的“收益止盈”和“溢价止盈”按百分比显示。字段口径对齐 VBA：`SellRatio` / `sell_ratio` 对应收益止盈，`TakeProfitPrice` / `take_profit_price` 对应溢价止盈，`AddPremiumLimit` / `add_premium_limit` 对应补仓溢价限制。
- 顶部“总资产（CNY）”小曲线优先来自真实 `account_replay_snapshot.total_assets` 快照；只有快照表为空时，才回退真实 `account_replay_state.total_assets` 历史记录。保存账户回放结果时会追加轻量快照，但只有总资产、总盈亏、未实现盈亏、现金、本金或估值完整性发生真实变化才写入；仅刷新时间变化不追加快照，金额变化达到 0.01 会保留，最多保留最近 500 条。
- 顶部“持仓盈亏”小曲线优先来自真实 `account_replay_snapshot.total_pnl`，无总盈亏时使用 `total_unrealized_pnl`；只有快照表为空时，才回退 `account_replay_state` 同名字段。曲线绘制前压缩连续重复值，并把真实变化点序列均匀铺展为迷你趋势图；没有真实点显示空态，单点显示“数据不足”，不补点、不复制最后值、不用刷新时间生成节点、不生成假波动、随机曲线、插值曲线或写死曲线。
- 顶部“今日盈亏”显示金额和百分比。金额优先使用回放持仓真实 `position_replay_state.daily_pnl` 汇总；缺失时使用当日快照 `total_pnl` 差额；仍缺失时使用 `total_assets` 差额并剔除当日 `入金` / `出金` 净现金流，避免把入金当盈利或把出金当亏损。百分比按今日盈亏金额 / 今日起始总资产计算，分母不可用时使用最新总资产扣除今日盈亏后的金额；数据不足时显示 `--`。
- 顶部卡片按 A 股显示口径着色：今日盈亏、持仓盈亏金额和持仓盈亏率为正数时红色，为负数时绿色，0 或无值为中性色。
- 狙击资金池卡片显示“可用档位”。模块 4 策略派生表未生成前显示“待策略模块”，不写死档位份数。
- 策略指令和 T1-T6 前置状态由模块 4 策略派生表生成；OTCMap A/C 拆单、最终委托量化和券商下单仍不实现。

## 策略配置百分比字段

策略配置页支持以下百分比字段输入：

- 收益止盈：保存到 `sell_ratio`。
- 溢价止盈：保存到 `take_profit_price`。
- 补仓溢价限制：保存到 `add_premium_limit`。

输入和存储规则：

- 输入 `40%`、`40`、`0.40` 都按 40% 处理，数据库统一保存 `0.40`。
- 输入 `8%`、`8`、`0.08` 都按 8% 处理，数据库统一保存 `0.08`。
- 输入 `2%`、`2`、`0.02` 都按 2% 处理，数据库统一保存 `0.02`。
- 空值允许保存为空；UI 空态显示为空或 `--`。
- 非法输入如 `abc%` 会在状态栏提示字段错误，不保存错误值，不让程序退出。

## 人工验收步骤

1. 执行 `dotnet run --project .\CrossETF.Terminal.UiShell.Reference.csproj` 启动程序。
2. 打开“本地数据手动录入”窗口。
3. 在 TradeLog 页录入一条 `CASH` / `入金` 记录并保存，确认底部状态显示已保存并完成账户回放。
4. 录入一条场内 ETF `买入` 记录并保存。
5. 回到主界面，确认账户状态面板显示回放现金、最新 TradeLog ID、持仓成本、总资产或估值不完整状态。
6. 确认 ETF 表格显示该策略的持仓数量、成本和本金占比。
7. 关闭程序并重新启动，确认刚才的回放状态和持仓仍能读回。
8. 故意录入一条现金余额不匹配的 TradeLog，保存后确认 UI 显示“财务异常”，`runtime_log` 写入明确错误。
9. 修正该 TradeLog 后保存，确认状态恢复为正常或估值不完整。
10. 验证三行场景：`CASH 入金 100000`、`159941 买入 1000`、`CASH 入金 5000`，即使不填或填错现金余额，保存后现金余额也应推演为 `100000`、`99000`、`104000`。
11. 验证买入/卖出输入价格和数量后，金额按 `price * quantity` 自动计算到分。
12. 验证小数数量保存读回后不丢失。
13. 删除中间 TradeLog 后保存，确认数据库记录数减少、被删 ID 不再出现，回放现金按剩余记录重算。
14. 确认 UI 没有新增手动刷新按钮。
15. 确认没有模拟行情、模拟交易、假高点、假曲线或策略下单入口。
